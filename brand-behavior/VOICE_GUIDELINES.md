# Voice Guidelines

> **Part of:** [`../BRAND_BEHAVIOR_SYSTEM.md`](../BRAND_BEHAVIOR_SYSTEM.md)
> **Not a duplicate of [`../advertising-system/copywriting.md`](../advertising-system/copywriting.md).** That document governs marketing copy — headlines, campaign lines, ad scripts. This document governs the voice everywhere marketing doesn't reach: product UI, AI replies, system-generated text, support scripts. The same five voice adjectives from `advertising-dna.md §11` apply in both places; this document specifies how they survive translation into interfaces and automated systems, where there's no creative director reviewing every line before it ships.

---

## The Same Five Adjectives, Held to a Harder Standard

**Warm. Clear. Grounded. Guiding. Respectful.** (`advertising-dna.md §11`.) In an ad, these are checked by a human before publish. In a product, they have to survive being generated by a template, an AI model, or a rushed engineer writing a fallback error string at 11pm — which means the standard has to be more mechanical, not more relaxed.

## Voice in the First Person, Never the Corporate Third Person

Tuba's product voice speaks as "we" to the user's "you" — never "Tuba Real Estate Platform Inc. has processed your request." First person makes every automated message harder to write coldly by construction; third-person corporate voice is banned outright in any customer-facing string.

## No Jargon Unless the User Used It First

Terms like "FAL," "REGA," "escrow," or "yield" are used only after the platform has confirmed the user is fluent in them (an investor's own language, a broker's own terminology) — otherwise, the plain-language version leads, with the technical term offered as a secondary detail, never the only way something is explained.

## Sentence Length Discipline

Product and system copy defaults to short, direct sentences — one idea per sentence, no compound clauses stacked to sound more official. A confirmation message, an error string, or an AI reply that reads like a legal notice has already failed the voice check regardless of its content being accurate.

## Voice Adapts in Register, Never in Values

Exactly mirroring `advertising-dna.md §9`: a WhatsApp message is more casual than a formal document-request email, and an AI chat reply is more conversational than a loan-decision letter — but warmth, clarity, and respect apply at every register, with no channel exempted as "just transactional, tone doesn't matter here."

## The Five Words Tuba's Product Voice Never Uses

1. **"Simply"** / **"Just"** before an instruction — if it were simple, the user wouldn't need the instruction; using the word implies the user should have already known.
2. **"Unfortunately"** as a stock lead-in to bad news — name what happened plainly instead of performing regret through a filler word.
3. **"Kindly"** — a translation-register formality that reads stiff in both English and natural Arabic; direct requests are simply direct.
4. **"Error"** as a user-facing label with no further explanation — always paired with what happened and what to do.
5. **"Oops!"** or manufactured cheerfulness around a genuine problem — see `ERROR_BEHAVIOR.md`'s tone standard.

## Arabic Voice Is Not a Translated Voice

Every rule above is written natively in Arabic, not translated from an English source string, per `advertising-dna.md §9` and `brand-experience/README.md`'s own practice. Arabic product copy is reviewed against the same five adjectives independently — a string that sounds warm in English and stiff in Arabic has failed, even if the translation is technically accurate.

## The One-Line Test

Before any system-generated string ships: **read it out loud as if a person, not a system, is saying it directly to a friend.** If it sounds like something a form would say, rewrite it until it sounds like something a person would say.
