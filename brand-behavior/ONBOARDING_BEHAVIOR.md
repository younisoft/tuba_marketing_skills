# Onboarding Behavior

> **Part of:** [`../BRAND_BEHAVIOR_SYSTEM.md`](../BRAND_BEHAVIOR_SYSTEM.md)
> **Covers the moment map's first three beats:** before Tuba, first visit, first search. This is the highest-leverage moment in the entire behavior system — a person has no prior experience of the brand to draw on yet, so behavior alone, in the first few minutes, decides whether "Everyone's Tuba Is Different" reads as true or as a slogan.

---

## Before Tuba

A person arriving from an ad or a search result carries an expectation set by that touchpoint. Onboarding's first job is to honor it exactly — if the ad promised guidance, the first screen guides; it does not open with an account-creation wall before delivering a single moment of the value promised. `advertising-dna.md §12`'s desired first-exposure feeling — *"this one seems different, warmer"* — must be true within the first screen, not the first session.

## First Visit

**What the customer should feel:** curious, not assessed. Like they've walked into a place that wants to understand them before it wants their information.

**What Tuba should do:** ask the smallest number of questions that produce a genuinely useful first result — never a long qualification form before any value is shown. Where account creation is required, it happens *after* a person has seen enough to want to save what they found, not before.

**What should never happen:** a generic, one-size welcome screen with no memory of what brought the person here (the ad they clicked, the search they typed). A cold start is acceptable; a start that ignores available context is not.

## First Search

**What the customer should feel:** heard, even with very little information given yet.

**What Tuba should do:** treat the first search as the beginning of a relationship, not a one-off query — this is where `SEARCH_BEHAVIOR.md`'s uncertainty-friendly, question-asking posture matters most, since a first-time user often doesn't yet know how to articulate what they want.

**What should never happen:** a results page with no explanation of why these results, forcing the person to either trust the platform blindly or start manually re-filtering everything, which reads as the platform not actually having understood the query at all.

## Pacing Onboarding to Experience Level

Per `EXPERIENCE_PRINCIPLES.md` §2, onboarding depth is not fixed — a returning investor who has clearly done this before is not walked through the same explanatory beats as someone visibly encountering Saudi real estate process for the first time. Signals (account type selected, language of query, prior platform history if any) adjust how much is explained versus assumed, without ever making the assumption feel presumptuous if it's wrong — a single, easy way to say "actually, walk me through this" is always visible.

## Progressive, Not Front-Loaded

Onboarding never asks for information before it's needed. Budget, timeline, and family context are gathered as they become relevant to the specific thing the person is doing next, not collected upfront in a single long intake form that makes the platform feel like paperwork before it's felt like help.

## What Onboarding Must Never Do

- Never gate basic browsing behind full registration — value is shown before commitment is asked for.
- Never use dark-pattern defaults (pre-checked marketing consent, auto-selected higher-commitment account types) to improve conversion metrics at the expense of an honest choice.
- Never make a first-time user feel like their inexperience is visible or judged — no copy or flow that assumes prior real-estate transaction knowledge without offering a plain-language alternative path.
