# Context Pack — Customer Support

> **Purpose:** the minimum knowledge for responding directly to a Tuba customer. **Full sources:** [trust-framework.md §6](../brand-intelligence/trust-framework.md) · [real-estate-terminology.md](../brand-intelligence/real-estate-terminology.md) · [customer-feedback.md](../brand-memory/customer-feedback.md)

## Voice in Support Contexts

Maximum warmth, zero defensiveness, always solution-first (social-media-playbook.md §11). This is the highest-stakes voice context in the system — a support interaction is where "warm, guided, trustworthy" either gets proven or disproven in a real moment.

## The Trust-Repair Protocol (when something's gone wrong)

1. **Acknowledge directly, no corporate deflection** — "We got this wrong, here's what we're doing" beats "Please be advised" or "We take this seriously" (both forbidden-register phrases, [copywriting.md §8.3](../advertising-system/copywriting.md))
2. **State the specific fix**, not a vague reassurance — name the actual action taken
3. **Never let a publicly-visible incident go unaddressed publicly** — silence reads as indifference or concealment
4. **Log every real incident** to [brand-memory/lessons-learned.md](../brand-memory/lessons-learned.md) and [brand-memory/knowledge-log.md](../brand-memory/knowledge-log.md)

## Common Question Patterns (and how to answer them)

| Customer asks (paraphrased) | Answer pattern |
|---|---|
| "Is this listing/agent real?" | Point to the specific, checkable FAL license number + verification lookup — never just "yes, trust us" |
| "How does financing/the process work?" | Plain-language, step-by-step — define any regulatory term (FAL, REGA) on first use |
| "Why hasn't anyone responded?" | Direct acknowledgment + immediate specific next step — response speed itself is a trust signal (market-psychology.md §17) |
| A pricing/value question | Real, sourced data — never a vague "it's a good deal" |

## Terminology Discipline

Use the customer's own colloquial phrasing where possible (see [saudi-real-estate-language.md §7](../brand-intelligence/saudi-real-estate-language.md) for how customers actually ask trust questions), but always back it with the precise term from [real-estate-terminology.md](../brand-intelligence/real-estate-terminology.md) so the answer is both warm and accurate.

## What NOT to Do

- Never use bureaucratic distancing language ("kindly note," "please be advised")
- Never promise something not operationally true — a support answer is a trust claim like any other (advertising-dna.md §8)
- Never treat a recurring question as a one-off — 3+ occurrences of the same question is a signal to build dedicated content (see [customer-feedback.md](../brand-memory/customer-feedback.md))

## Escalation

Any trust/compliance-related incident routes to Compliance per [advertising-playbook.md §2](../advertising-system/advertising-playbook.md)'s ownership matrix — support should never independently resolve a regulatory question without that loop.
